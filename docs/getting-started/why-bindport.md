# Why BindPort

BindPort exists because local development usually starts simple and then gets
messy:

- one app becomes several apps;
- one checkout becomes multiple worktrees;
- one fixed port becomes a spreadsheet of reserved ports;
- one local URL becomes branch-specific URLs;
- one dev script becomes a mix of framework flags, env vars, proxy snippets,
  cleanup commands, and dashboard tabs.

BindPort gives that local state one explicit owner without taking over the
whole local stack.

## The Core Problem

Most development tools treat ports as process-local details. A web server knows
which port it bound. A proxy knows which upstream it should route to. A package
script knows which flags it passed. A browser tab knows the URL it opened.

Those pieces often do not share a durable source of truth.

That creates common failure modes:

- another branch starts on the same port and breaks the first one;
- a service moves ports and the proxy still points at the old port;
- a dashboard or AI agent guesses `localhost:3000` because there is no better
  discovery mechanism;
- `.env.local` or proxy config is regenerated by hand and drifts from reality;
- cleanup is risky because nobody knows whether a stale-looking entry is still
  owned by a live process.

BindPort's registry is the shared source of truth for local route state.

## The BindPort Shape

BindPort is intentionally small in scope:

1. Resolve a project/service identity.
2. Allocate or reuse a stable port for that identity.
3. Run or reserve the service.
4. Record the state in a local SQLite registry.
5. Expose that state through CLI, JSON, dashboard, templates, and hooks.

It does not become the reverse proxy, certificate authority, DNS manager,
secret manager, or process supervisor. That boundary matters because teams
already have strong opinions about those tools.

## Before And After

Without BindPort:

```sh
PORT=3000 npm run dev
```

The service works until another service or worktree also wants `3000`. Scripts,
proxy files, browser bookmarks, and agents all need side knowledge.

With BindPort:

```sh
bindport run web
bindport open web --print
bindport status --json
```

The service gets a stable assigned port, route metadata is recorded, and other
tools can discover the current URL.

## Where It Helps Most

Monorepos:

- service names become explicit;
- each package can have its own command, args, route URL, and health check;
- generated proxy or env files can be scoped per service.

Branch and worktree workflows:

- branch labels can become route hostnames;
- separate worktrees avoid fighting over one fixed port;
- stale entries can be cleaned after branches are removed.

Remote development boxes:

- the dashboard can be served with auth on `0.0.0.0`;
- services can bind to the right host while route metadata remains explicit;
- agents can discover URLs without asking the user to paste ports.

Existing proxy setups:

- Traefik or another proxy remains the proxy;
- BindPort renders text files into a watched directory;
- hooks can reload external tools after reviewed local trust.

AI-assisted development:

- agents can run known project commands instead of hardcoding ports;
- `status --json` gives machine-readable service and route state;
- agent instructions can prohibit local-only file edits and hook trust changes.

## What It Deliberately Does Not Do

BindPort does not:

- bind `80` or `443`;
- install certificates;
- mutate DNS or `/etc/hosts`;
- run a root daemon;
- overwrite unowned generated files;
- trust hooks from checked-in config;
- kill active processes as a cleanup shortcut.

Those choices keep BindPort useful in existing local stacks without making it
the most privileged part of the machine.

## When Not To Use It

BindPort is probably unnecessary when a project has one service, one checkout,
no local proxy, no generated integration files, and no need for machine-readable
service discovery.

It becomes useful when the local environment needs stable coordination across
services, worktrees, scripts, dashboards, proxies, and agents.
